# Skill: Verify History

## Task
AI가 작성한 포스트 초안의 내러티브가 실제 경험과 일치하는지 검증한다.
AI는 코드와 문서를 기반으로 "그럴듯한 이야기"를 만들지만, 실제 경험과 다를 수 있다.
이 스킬은 초안의 핵심 스토리를 사용자에게 하나씩 확인받는 체크포인트다.

## Why This Exists
- AI가 탐색한 코드/MEMORY.md로 추론한 "사건"이 실제와 다를 수 있음
- 예시: "npm create vite 실수" → 실제는 "두 AI 에이전트 간 소통 없이 작업시킨 사고"
- 기술적 사실은 맞아도 맥락/감정/동기가 틀릴 수 있음
- 비개발자 관점의 체감 경험은 코드에서 읽어낼 수 없음

## Input
- Phase 2에서 작성된 MDX 초안
- solution-explorer의 분석 리포트

## Process

### Step 1: 내러티브 추출
초안에서 "사실 주장"이 포함된 스토리를 모두 추출한다:

- **사건(Event)**: "이런 일이 있었다" — 시행착오, 사고, 발견
- **동기(Motivation)**: "이래서 이걸 선택했다" — 기술 결정 이유
- **감정(Emotion)**: "이때 이런 기분이었다" — 좌절, 성취감, 놀라움
- **수치(Number)**: "X시간 걸렸다", "Y% 개선됐다" — 정량적 주장
- **인물/도구(Actor)**: "누가/무엇이 이걸 했다" — AI 도구, 팀원, 서비스

### Step 2: 사용자 검증 (★ 체크포인트 2.5)
추출한 스토리를 표로 정리하여 사용자에게 제시:

```
| # | 유형 | 초안 내러티브 | 확인 |
|---|------|-------------|------|
| 1 | 사건 | "npm create vite로 파일이 덮어써졌다" | ✅/❌ |
| 2 | 동기 | "API가 너무 많아서 MCP를 선택했다" | ✅/❌ |
| 3 | 수치 | "복구에 반나절 걸렸다" | ✅/❌ |
| 4 | 감정 | "세션 버그에 좌절했다" | ✅/❌ |
| 5 | 인물 | "혼자 개발했다" | ✅/❌ |
```

각 항목에 대해:
- ✅ 맞으면 패스
- ❌ 틀리면 실제 히스토리를 사용자에게 받아서 수정

### Step 3: 톤 검증
사용자의 실제 말투가 반영됐는지 확인:
- 사용자가 수정해준 문장의 톤앤매너를 분석
- `;;;`, `ㅠㅠ`, 자조적 유머, 사람 비유 등 사용자 고유 화법 확인
- 너무 "기술 블로그"스러운 문체가 없는지 확인

### Step 4: 수정 반영
- ❌ 항목을 사용자가 알려준 실제 히스토리로 교체
- 톤앤매너도 함께 조정
- 수정 완료 후 사용자에게 재확인

## Output
- 검증 완료된 MDX 초안 (Phase 3: Media로 전달)
- 새로 알게 된 히스토리 → `sot/writing-style-guide.md`에 톤 패턴 추가 (필요시)

## 검증 기준
- 모든 사건/동기/수치가 사용자 확인됨 (❌ 항목 0개)
- 사용자 고유 톤앤매너가 최소 3곳 이상 반영됨
- "AI가 추측한 느낌"이 남아있지 않음
